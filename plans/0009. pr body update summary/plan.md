# Plan: GitHub Action PR Body Update Summary

> Source PRD: [plans/0009. pr body update summary/prd.md](/Users/mishankov/dev/updtr/plans/0009.%20pr%20body%20update%20summary/prd.md)

## Architectural decisions

Durable decisions that apply across all phases:

- **Feature boundary**: this is a follow-up to the existing GitHub Action. Branch management, commit flow, and pull request lifecycle stay in the action layer.
- **Result source**: pull request body content comes from structured `updtr apply` results, not from git diff parsing and not from scraping rendered CLI output.
- **Publishing contract**: the action should send the same reviewer-facing payload on both pull request creation and pull request update.
- **Body scope**: the first version summarizes applied updates only. It does not attempt to render blocked candidates or full planning diagnostics.
- **Rendering model**: the PR body is deterministic Markdown with stable section ordering and stable update ordering inside each target.
- **Review shape**: the generated body should include a concise summary plus per-target applied-update sections, with direct versus indirect labeling and vulnerability context when available.
- **Failure contract**: no-op and failed runs must not publish misleading PR body content. Body generation only happens on successful changed runs that reach PR publication.
- **Action interface**: the feature should not introduce new mandatory action inputs or early template customization.
- **Testing focus**: tests should emphasize observable contracts across result transport, Markdown rendering, runtime orchestration, and GitHub API requests.

## Phase 1: Structured Apply Results Reach PR Creation

**User stories**: 1, 2, 3, 8, 14, 17, 18, 21

### What to build

Create the first end-to-end slice that carries structured update results from the `updtr apply` path into the action runtime and uses them to populate the PR body on new pull requests. This slice should replace the current placeholder body with a minimal but real summary derived from the applied updates that were actually committed.

### Acceptance criteria

- [ ] The action runtime has access to structured apply results for changed runs instead of treating `updtr apply` as write-only execution.
- [ ] New pull requests created by the action include a generated Markdown body derived from applied updates.
- [ ] The generated body includes at least the applied dependency name and version transition for each applied update.
- [ ] The current fixed placeholder PR body text is removed from the create path.
- [ ] No-op runs continue to skip PR creation and therefore do not render or publish a PR body.
- [ ] Tests cover apply-result transport into the action runtime and PR creation requests that include a generated `body` field.

## Phase 2: Existing PRs Refresh Title and Body Together

**User stories**: 9, 10, 14, 15, 16, 20, 21

### What to build

Extend the repeated-run path so existing managed pull requests are updated with the latest generated body, not just a refreshed title. This slice makes the feature reliable for scheduled runs by ensuring that the open PR always matches the latest branch state and that body publication remains gated behind successful changed runs.

### Acceptance criteria

- [ ] Repeated runs against an existing managed PR update the PR body as well as the title.
- [ ] The update path uses the same body-generation contract as the create path.
- [ ] Failed runs do not publish partial or stale body content.
- [ ] No-op reruns remain successful and do not call the PR update path.
- [ ] The generated body is deterministic for equivalent apply results so repeated runs do not cause noisy body churn.
- [ ] Tests cover PR update requests that include both title and body, plus no-op and failed-run gating behavior.

## Phase 3: Reviewer-Friendly Summary and Target Grouping

**User stories**: 4, 5, 11, 23, 24, 25, 26

### What to build

Upgrade the minimal body into a reviewer-friendly Markdown summary. The body should start with concise counts, then group applied updates by target so maintainers of multi-target repositories can quickly see what changed and where. Only targets with actual applied updates should appear in the body.

### Acceptance criteria

- [ ] The generated PR body contains a concise summary section with counts such as changed targets and applied updates.
- [ ] Applied updates are grouped into per-target sections.
- [ ] Each rendered update clearly shows whether it is a direct or indirect dependency change.
- [ ] Targets with no applied updates are excluded from the applied-update summary.
- [ ] The body wording makes it clear that the PR was generated by `updtr`.
- [ ] Tests cover stable target grouping, stable update ordering, summary counts, and exclusion of empty targets.

## Phase 4: Vulnerability and Warning Context

**User stories**: 6, 7, 12, 13, 19, 20

### What to build

Add the reviewer context that makes dependency PRs more actionable without turning the body into a log dump. This slice surfaces vulnerability-remediation context and action-relevant warnings in concise sections attached to the applied-update summary while keeping the Markdown deterministic and readable.

### Acceptance criteria

- [ ] When applied updates include vulnerability metadata, the PR body surfaces that context in a concise reviewer-readable form.
- [ ] Advisory identifiers are included when available.
- [ ] Action-relevant warnings from the apply result are surfaced in the body without relying on raw command output.
- [ ] The rendered body omits empty vulnerability and warning sections cleanly.
- [ ] The body remains concise and deterministic even when many updates are present.
- [ ] Tests cover vulnerability rendering, advisory rendering, warning rendering, and omission behavior when those fields are absent.

---

## Phase 5: Consumer Contract and Hardening

**User stories**: 14, 15, 16, 20, 22

### What to build

Harden the feature as part of the public action contract. This slice aligns documentation, examples, and integration tests with the new PR body behavior while preserving the intentionally small action interface and the existing no-op and failure semantics.

### Acceptance criteria

- [ ] Consumer-facing documentation explains that update details are published into the PR body for changed runs.
- [ ] The documented action interface remains small and does not add mandatory customization inputs for this feature.
- [ ] Integration coverage verifies the body-generation behavior through the action runtime boundary rather than only unit-level formatting tests.
- [ ] The feature preserves existing no-op semantics and explicit failure behavior across the published action contract.
- [ ] Documentation and tests stay aligned with the generated-body format and create-or-update behavior.
